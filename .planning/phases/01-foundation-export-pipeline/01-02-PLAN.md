**Structure:**

```typescript
import { type NextRequest } from 'next/server'
```

**Auth guard (per INF-01, D-07):**
```typescript
const auth = requireAdmin(req)
if (auth instanceof Response) return auth
```

**Streaming setup (per EXP-06):**
- Create a `PassThrough` stream
- Create archiver instance: `archiver('zip', { zlib: { level: 6 } })`
- Pipe archiver into PassThrough
- Handle archiver `error` and `warning` events (destroy PassThrough on error)
- Start `buildExport(archive)` asynchronously (do NOT await — the response streams immediately)
- Return `new Response(Readable.toWeb(pass) as ReadableStream, { headers })` with:
  - `Content-Type: application/zip`
  - `Content-Disposition: attachment; filename="snackspot-export-${Date.now()}.zip"`

**`buildExport(archive)` async function:**

This function queries all 18 tables, serializes them, appends photos, writes manifest, and finalizes.

**Table export order and serialization (per D-04, D-06, D-08, D-09, D-10):**

For each table, query with `db.{model}.findMany()` (except Place), serialize to JSON, and append to archive as `data/{tablename}.json`.

1. **users** — `db.user.findMany()` — serialize all fields as-is. Note: passwordHash IS included (intentional for full backup, per RESEARCH security domain).

2. **places** — Use `$queryRaw` (per EXP-03, D-09):
   ```typescript
   const rawPlaces = await db.$queryRaw<Array<{
     id: string; name: string; address: string;
     lat: number; lng: number;
     created_at: Date; updated_at: Date;
   }>>`
     SELECT id, name, address,
       ST_Y(location::geometry) AS lat,
       ST_X(location::geometry) AS lng,
       created_at, updated_at
     FROM places
   `
   ```
   Then map to `{ id, name, address, location: { lat, lng }, createdAt, updatedAt }`. Note the column names from raw SQL are snake_case — convert to camelCase to match Prisma convention.

3. **reviews** — `db.review.findMany()` — convert all Decimal fields (per EXP-04, D-08):
   ```typescript
   reviews.map(r => ({
     ...r,
     rating: Number(r.rating),
     ratingTaste: Number(r.ratingTaste),
     ratingValue: Number(r.ratingValue),
     ratingPortion: Number(r.ratingPortion),
     ratingService: r.ratingService === null ? null : Number(r.ratingService),
     ratingOverall: Number(r.ratingOverall),
   }))
   ```

4. **photos** — `db.photo.findMany()` — variants and metadata are Json type, export as-is (per D-10).

5. **reviewPhotos** — `db.reviewPhoto.findMany()`
6. **comments** — `db.comment.findMany()`
7. **badges** — `db.badge.findMany()`
8. **userBadges** — `db.userBadge.findMany()`

For each table, append: `archive.append(JSON.stringify(serialized, null, 2), { name: 'data/{tablename}.json' })`

Use these exact file names in `data/`: `users.json`, `places.json`, `reviews.json`, `photos.json`, `review-photos.json`, `comments.json`, `badges.json`, `user-badges.json`, `review-likes.json`, `favorites.json`, `reports.json`, `moderation-actions.json`, `notifications.json`, `notification-preferences.json`, `review-mentions.json`, `review-tags.json`, `blocked-words.json`, `flagged-comments.json`

**Photo streaming (per EXP-05, D-05, D-11, D-12):**

After all table JSONs are appended, stream photos from MinIO:
```typescript
const allPhotos = await db.photo.findMany({ select: { id: true, storageKey: true } })
let photosExported = 0
let photosSkipped = 0

for (const photo of allPhotos) {
  try {
    const stream = await minioClient.getObject(BUCKET, photo.storageKey)
    archive.append(stream, { name: `photos/${photo.storageKey}` })
    photosExported++
  } catch {
    photosSkipped++
    console.error(`Export: skipped missing photo ${photo.id} (key: ${photo.storageKey})`)
  }
}
```

Use sequential `for...of` with `await` — no p-limit, no parallel fetches.

**Manifest (per EXP-08, INF-03, D-03):**

After all tables and photos are appended, create and append the manifest:
```typescript
interface ExportManifest {
  schemaVersion: number
  exportedAt: string
  tables: string[]
  counts: Record<string, number>
  photosCount: number
  photosSkipped: number
}

const manifest: ExportManifest = {
  schemaVersion: 1,
  exportedAt: new Date().toISOString(),
  tables: [ /* list of 18 table file names */ ],
  counts: { users: users.length, places: places.length, /* ... all 18 */ },
  photosCount: photosExported,
  photosSkipped: photosSkipped,
}
archive.append(JSON.stringify(manifest, null, 2), { name: 'manifest.json' })
```

**Finalize:**
```typescript
await archive.finalize()
```

Call `archive.finalize()` AFTER all appends are complete. This is the last line of `buildExport()`. The `await` ensures finalize completes before the function resolves, but the response stream has already started flowing.

**Error handling:**
- Wrap the entire `buildExport` body in try/catch
- On error, call `archive.abort()` and `pass.destroy(err)` to terminate the stream
- Individual photo errors are caught and logged (skip + continue, per Claude's Discretion)

**Important implementation notes:**
- `export const runtime = 'nodejs'` MUST be at module top level (Edge runtime does not support node:stream)
- Do NOT use `db.refreshToken`, `db.passwordResetToken`, or `db.emailVerificationToken` — these are the excluded token tables (D-07)
- Track record counts during export for the manifest (store the `.length` of each array)
- The `buildExport` call in the GET handler must NOT be awaited — it runs asynchronously while the response streams. Errors are caught and forwarded to the PassThrough stream via `.catch((err) => pass.destroy(err))`.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> GET /api/export | Unauthenticated/unauthorized requests must be rejected before any DB access |
| DB -> ZIP response body | Sensitive data (password hashes, user emails) flows into downloadable archive |
| MinIO -> ZIP response body | Photo binaries from object storage flow into downloadable archive |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | GET /api/export | mitigate | `requireAdmin(req)` guard returns 401/403 before any DB query executes |
| T-01-02 | Information Disclosure | User.passwordHash in export | accept | Intentional for full backup fidelity. Export is ADMIN-only. Password hashes are bcrypt/argon2 — not reversible. Admin already has DB access. |
| T-01-03 | Information Disclosure | Token tables | mitigate | D-07 locked: RefreshToken, PasswordResetToken, EmailVerificationToken excluded from export. Code must not reference these models. |
| T-01-04 | Denial of Service | Concurrent export requests | accept | Admin population is 1-2 users. Export is behind admin auth. No rate limiting needed for v1. |
| T-01-05 | Tampering | Photo storageKey path traversal | mitigate | `photo.storageKey` comes from trusted DB (not user input in this request). Used directly as MinIO object key — no string concatenation with user input. |
| T-01-06 | Denial of Service | Large dataset memory spike | mitigate | Use streaming archiver (not in-memory buffer). Each table queried and serialized independently. Photos streamed from MinIO without buffering. |
</threat_model>
