## Summary

Phase 2 builds a single POST API endpoint (`/api/import`) that accepts a ZIP file, validates it, remaps all IDs, and inserts 18 tables in FK dependency order — all within a single Prisma transaction for atomicity. The ZIP parsing library is `unzipper` (already decided in STATE.md; aligns with project's existing `archiver` for export). All 18 tables already have `gen_random_uuid()::text` as a DB-level default for their primary key, eliminating the need for client-side ID generation. Place insertion requires raw SQL because of the PostGIS `geography(Point, 4326)` column. Every other table uses Prisma ORM methods.

The UI extends the existing Dutch export page with a file picker + upload button + inline summary report. The approach is synchronous within the HTTP request — no background jobs, no SSE, no polling.

**Primary recommendation:** Use `unzipper.Open.buffer()` to parse the ZIP in memory, validate first, then run all DB writes inside a single `db.$transaction(async tx => {...}, { timeout: 60000, maxWait: 10000 })`.

### Pattern 1: ZIP Parsing with unzipper.Open.buffer

**What:** Read the uploaded ZIP as a Buffer, open it with unzipper, then access individual entries by path.
**When to use:** When the ZIP is fully available in memory (Phase 2 — JSON-only ZIP, no photos).

```typescript
// Source: unzipper npm README (verified)
import unzipper from 'unzipper'

const formData = await req.formData()
const file = formData.get('file') as File
const buffer = Buffer.from(await file.arrayBuffer())
const directory = await unzipper.Open.buffer(buffer)

// Find a specific file by path
const manifestFile = directory.files.find(f => f.path === 'manifest.json')
if (!manifestFile) throw new Error('manifest.json missing')
const manifestContent = await manifestFile.buffer()
const manifest = JSON.parse(manifestContent.toString('utf-8'))
```

[CITED: unzipper npm README — `Open.buffer(buffer)` returns directory with `.files[]` array; each file has `.buffer()` Promise]

### Pattern 3: ID Remapping with Per-Table Maps

**What:** For each table, maintain a `Map<string, string>` from old export ID to new DB-generated ID.
**When to use:** Every table with its own `id` primary key AND every FK field on dependent tables.

```typescript
// Source: [ASSUMED] — standard pattern for data migration
type IdMap = Map<string, string>
const idMaps: Record<string, IdMap> = {
  users:  new Map(),
  places: new Map(),
  reviews: new Map(),
  photos: new Map(),
  comments: new Map(),
  badges: new Map(),
  // junction tables (UserBadge, ReviewLike, etc.) have compound PKs — no own ID map needed
}

// Example: User insert within transaction
const newUser = await tx.user.create({
  data: {
    email: record.email,
    username: record.username,
    // ... all fields EXCEPT id (DB generates a new one)
  },
  select: { id: true },
})
idMaps.users.set(record.id, newUser.id)  // oldId -> newId

// Example: Review insert with FK remapping
const newReview = await tx.review.create({
  data: {
    userId:  idMaps.users.get(record.userId)!,   // remapped
    placeId: idMaps.places.get(record.placeId)!, // remapped
    // ... other fields
  },
  select: { id: true },
})
idMaps.reviews.set(record.id, newReview.id)
```

### Pattern 4: PostGIS Place Insertion via Raw SQL

**What:** `Place.location` is `geography(Point, 4326)` — Prisma cannot write this via ORM; raw SQL required.
**When to use:** Every Place insert.

```typescript
// Source: confirmed from apps/web/app/api/v1/reviews/route.ts (existing pattern in codebase)
// and apps/admin/app/api/export/route.ts (export uses ST_Y/ST_X for reading)
const [{ id: newId }] = await tx.$queryRaw<Array<{ id: string }>>`
  INSERT INTO places (name, address, location, created_at, updated_at)
  VALUES (
    ${record.name},
    ${record.address},
    ST_SetSRID(ST_MakePoint(${record.location.lng}, ${record.location.lat}), 4326)::geography,
    ${new Date(record.createdAt)},
    ${new Date(record.updatedAt)}
  )
  RETURNING id
`
idMaps.places.set(record.id, newId)
```

**CRITICAL:** `ST_MakePoint(x, y)` takes longitude first (x), latitude second (y). Export serializes as `{ lat, lng }`. Swap order on re-insertion.

[VERIFIED: `apps/web/app/api/v1/reviews/route.ts` line 51-60 — identical INSERT pattern exists in codebase]

### Pattern 5: Prisma Transaction with Extended Timeout

**What:** Wrap all 18 table insertions in a single interactive transaction for atomicity (D-05).
**When to use:** The entire import write phase.

```typescript
// Source: Prisma 5 docs [ASSUMED — version confirmed as 5.22.0]
const result = await db.$transaction(async (tx) => {
  // all tx.model.create() and tx.$queryRaw() calls here
  return summaryReport
}, {
  timeout: 60000,   // 60 seconds — default 5000ms is too short for large imports
  maxWait: 10000,   // wait up to 10s to acquire a connection
})
```

**PgBouncer compatibility:** The admin app's `DATABASE_URL` in Docker resolves to `DATABASE_POOL_URL` (PgBouncer in transaction-pool mode). Prisma interactive transactions (`$transaction(callback)`) ARE compatible with PgBouncer transaction mode — a connection is held for the entire transaction duration, then released on commit/rollback. No incompatibility. [VERIFIED: confirmed from docker-compose.yml — `DATABASE_URL: ${DATABASE_POOL_URL:-${DATABASE_URL:?...}}`]

### Pattern 6: Duplicate Detection with Pre-loaded Lookup Maps

**What:** Before the transaction, load existing users and places to enable O(1) dedup lookups.
**When to use:** During the validation/pre-flight phase before any writes.

```typescript
// Source: [ASSUMED] — standard pattern
const existingUsers = await db.user.findMany({ select: { id: true, email: true } })
const userEmailMap = new Map(existingUsers.map(u => [u.email.toLowerCase(), u.id]))

const existingPlaces = await db.$queryRaw<Array<{id: string, name: string, address: string}>>`
  SELECT id, name, address FROM places
`
const placeKeyMap = new Map(existingPlaces.map(p => [`${p.name}::${p.address}`, p.id]))

// Usage in import loop:
const dupUserId = userEmailMap.get(record.email.toLowerCase())
if (dupUserId) {
  idMaps.users.set(record.id, dupUserId)  // remap to existing
  userStats.skipped++
  continue
}
```

### Pattern 7: Junction Table Insert with createMany + skipDuplicates

**What:** For tables with compound PKs and no own `id`, use `createMany` after FK remapping.
**When to use:** ReviewLike, Favorite, ReviewPhoto, UserBadge, ReviewMention, ReviewTag, NotificationPreferences.

```typescript
// Source: Prisma 5 docs [ASSUMED — createMany skipDuplicates available since Prisma 4]
const remappedLikes = reviewLikes.map(r => ({
  userId:   idMaps.users.get(r.userId)!,
  reviewId: idMaps.reviews.get(r.reviewId)!,
```

**What:** Parse and validate all data before any DB writes.

### Anti-Patterns to Avoid

- **Generating IDs client-side when DB defaults exist:** All 18 tables have `DEFAULT gen_random_uuid()::text` at the DB level (verified in `packages/db/migrations/001_init.sql`). Don't call `crypto.randomUUID()` for IDs — just omit `id` from the Prisma `create()` call and the DB generates it.
- **Using `ST_MakePoint(lat, lng)` (wrong order):** PostGIS `ST_MakePoint(x, y)` takes longitude as X, latitude as Y. Export serializes as `{ lat, lng }`. Import must call `ST_MakePoint(record.location.lng, record.location.lat)`.
- **Opening the transaction before validation:** Running D-04 validation inside the transaction wastes a connection and risks timeout on validation failures. Validate outside, then transact.
- **Using Prisma `create()` for Place:** `Place.location` is `Unsupported("geography(Point, 4326)")` — Prisma ORM cannot write this type. Must use `tx.$queryRaw`.
- **Default transaction timeout:** Prisma 5 default is 5000ms. A 18-table import of hundreds of records will exceed this. Always set `timeout: 60000`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ZIP parsing | Custom binary parser | `unzipper.Open.buffer()` | Central directory parsing, compression variants, entry streaming — complex binary format |
| ID uniqueness | Custom ID generator | DB `gen_random_uuid()` default via Prisma `create()` | Already present in all table DDL; no dependency needed |
| Transaction atomicity | Manual rollback logic with `try/catch/delete` | `db.$transaction(callback, {timeout})` | Prisma handles rollback, connection lifecycle, and error propagation |
| Compound dedup on junction tables | Custom unique check before each insert | `createMany({ skipDuplicates: true })` | Prisma 5 handles this natively with a single SQL UPSERT |

**Key insight:** Every "custom" solution here has been a source of bugs in data migration projects. The ecosystem tools handle edge cases (ZIP64 format, partial commits, PK collisions) that hand-rolled code misses.

## Common Pitfalls

### Pitfall 1: PostGIS Coordinate Order (ST_MakePoint)

**What goes wrong:** `ST_MakePoint(lat, lng)` silently swaps all geolocation data — places appear in the ocean or wrong hemisphere.
**Why it happens:** PostGIS convention is `(x, y) = (longitude, latitude)`. The export JSON stores `{ lat, lng }`. Developers intuitively write `ST_MakePoint(lat, lng)`.
**How to avoid:** Always `ST_MakePoint(record.location.lng, record.location.lat)`.
**Warning signs:** Place queries return zero results; MapLibre shows places far from expected region.

### Pitfall 2: Prisma Transaction Timeout with Large Datasets

**What goes wrong:** Import of a large instance (1000+ records) silently aborts mid-transaction with a Prisma timeout error.
**Why it happens:** Prisma 5 default `timeout` is 5000ms. Large imports take 10-60+ seconds.
**How to avoid:** Always specify `{ timeout: 60000, maxWait: 10000 }` on the transaction options.
**Warning signs:** Import returns a 500 error; Prisma logs show `Transaction already closed`.

### Pitfall 3: Optional FK Fields and Missing idMap Entries

**What goes wrong:** `Notification.actorId`, `Report.reviewId`, `FlaggedComment.reviewedBy` are optional (`?`). A mapping lookup returns `undefined` for null fields, which Prisma may interpret as `undefined` (omit field) or throw.
**Why it happens:** `idMaps.users.get(null)` returns `undefined`; JavaScript `Map.get(undefined)` also returns `undefined`. These need explicit null-pass-through.
**How to avoid:** Write remap helpers that short-circuit on null/undefined:
```typescript
const remap = (map: IdMap, oldId: string | null): string | null =>
  oldId == null ? null : (map.get(oldId) ?? null)
```
**Warning signs:** TypeScript errors at `tx.notification.create()`; runtime `null !== undefined` bugs.

### Pitfall 4: ReviewMention Has Three FK References to Different Tables

**What goes wrong:** Forgetting that `ReviewMention` has `reviewId`, `mentionedUserId`, AND `mentionedByUserId` — all need remapping from different maps.
**Why it happens:** Most tables have 1-2 FKs; ReviewMention has 3, two of which point to the users map.
**How to avoid:** When processing `review-mentions.json`, apply:
```typescript
reviewId:          idMaps.reviews.get(r.reviewId)!
mentionedUserId:   idMaps.users.get(r.mentionedUserId)!
mentionedByUserId: idMaps.users.get(r.mentionedByUserId)!
```
**Warning signs:** FK constraint violation on `review_mentions` insert; TypeScript error on missing map.

### Pitfall 5: Notification.link Contains Stale Old IDs

**What goes wrong:** After import, notification links (e.g., `/review/old-id-123`) point to non-existent resources because IDs changed.
**Why it happens:** `Notification.link` is a plain string — we import it as-is. IDs inside URLs are not remapped.
**How to avoid:** Accept this as a known limitation of Phase 2. The notification data integrity (userId, reviewId FK) is preserved; only the link string is stale. Document this in the summary report if desired.
**Warning signs:** Admin users clicking notification links get 404s after import. Expected — not a bug.

### Pitfall 6: Photo Records Import Without MinIO Files

**What goes wrong:** Phase 2 imports `photos.json` records (metadata) but skips the `photos/` directory in the ZIP. The `Photo` DB records exist but MinIO has no corresponding files. The web app serves broken image URLs.
**Why it happens:** Photo file import is Phase 3. This is intentional.
**How to avoid:** Accept this state. The `moderationStatus` will be PENDING on imported photos with no MinIO file. Phase 3 will upload the files. Document this in the import summary UI.
**Warning signs:** Broken images in the web app after a Phase 2 import. Expected until Phase 3 runs.

### Pitfall 7: Badge Dedup Not Covered by Explicit Decisions

**What goes wrong:** Badge records use `slug` as a `@unique` field. If the same badge slugs exist in both source and target, inserting them will throw a unique constraint violation.
**Why it happens:** D-07/D-08 only mention User and Place dedup. Badge dedup was not explicitly decided.
**How to avoid:** Treat Badge dedup the same as User/Place: check `slug` uniqueness; skip if exists; remap old badgeId to existing badge's id. [ASSUMED — natural extension of D-09 pattern]
**Warning signs:** Unique constraint error on `badges.slug` during import.

## Open Questions

1. **Idempotency: Can re-running the same archive really produce zero new records?**
   - What we know: D-11 says new IDs are generated (no source ID preserved). Users and Places are deduped. D-10 says child records of dup users are remapped (not skipped).
   - What's unclear: If User A is dup (maps to existing), Review B by User A will still be inserted as a new record with the existing userId. On second run, Review B is inserted AGAIN — no uniqueness check exists for Review.
   - Recommendation: Clarify with user whether success criterion (5) means "user+place level" idempotency, or full idempotency requiring source ID tracking. The most pragmatic interpretation: "second run of the same archive on a target that was populated entirely by the first run of that archive produces zero new records for User and Place (which are deduped), and therefore zero new records for any table because all Users and Places resolve as duplicates and D-10 remapping means child records also skip." This interpretation DOES hold if ALL users in the archive become dups on second run.

2. **Photo dedup: what is the natural unique field?**
   - What we know: `Photo.storageKey` is `@unique`. D-07/D-08 cover User and Place. No explicit decision for Photo.
   - What's unclear: Should Photo dedup use `storageKey`? Or should Photos always be re-inserted?
   - Recommendation: Use `storageKey` as the dedup field for Photo, consistent with the D-09 "existing records preserved" principle. Map old photoId to existing photo's id.

3. **BlockedWord.id uses `@default(dbgenerated("gen_random_uuid()"))` — different schema from other tables**
   - What we know: BlockedWord uses a different Prisma ID syntax (`dbgenerated`) rather than `@default(cuid())`.
   - What's unclear: Does this affect how Prisma handles omitting `id` in `create()`?
   - Recommendation: Use Prisma `create()` without `id` — the DB-level UUID default handles it. Confirmed by `001_init.sql`: `id TEXT NOT NULL DEFAULT gen_random_uuid()::text`.
