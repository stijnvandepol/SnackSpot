# Phase 2: Import Pipeline (Relational) - Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Admins can upload an export ZIP archive via the dashboard and have all relational data (18 tables) merged into the target instance — with FK dependency ordering, ID remapping, duplicate detection, and a per-table summary report. Photo import is a separate phase (Phase 3).

</domain>

<decisions>
## Implementation Decisions

### Import UI & feedback
- **D-01:** Import section added to the existing Export/Import page (`apps/admin/app/dashboard/export/page.tsx`) — file picker + upload button below the existing export card
- **D-02:** During import, show a spinner/loading state on the button. On completion, display a per-table summary report inline (imported / skipped / errors per table)
- **D-03:** UI copy in Dutch, consistent with the existing export page

### Validation & error handling
- **D-04:** Pre-import validation: check manifest.json exists and `schemaVersion` matches current version (INF-03). Validate each JSON file parses correctly before any DB writes (IMP-02)
- **D-05:** On hard errors, abort without partial writes (IMP-09) — use a database transaction wrapping all table inserts so failure rolls back everything
- **D-06:** Validation errors and import errors surfaced in the summary report UI — no silent failures

### Duplicate detection
- **D-07:** User duplicates detected by email, case-insensitive (`LOWER(email)` comparison) (IMP-05)
- **D-08:** Place duplicates detected by exact name + exact address match (IMP-05)
- **D-09:** Duplicates are skipped, not overwritten — existing records preserved (IMP-06)
- **D-10:** Records referencing a duplicate (e.g., reviews by a duplicate user) are remapped to the existing record's ID, not skipped

### ID remapping
- **D-11:** All imported records get new IDs generated by the database. A mapping table (old ID -> new ID) is maintained in memory during import for FK rewriting (IMP-04)
- **D-12:** Compound-key junction tables (ReviewLike, Favorite, ReviewTag) have their composite FKs remapped using the same ID mapping (IMP-04)
- **D-13:** Tables imported in FK dependency order: independent tables first (User, Place, Badge, BlockedWord), then dependent tables (Review, Photo, Comment, etc.), then junction tables last (IMP-03)

### Import processing approach
- **D-14:** Single POST request to `/api/import` — the ZIP is parsed server-side. Response returns the full summary report as JSON
- **D-15:** Synchronous processing within the request (not a background job) — keeps implementation simple and the summary report is returned directly in the response

### Claude's Discretion
- Exact FK dependency ordering of all 18 tables
- ZIP parsing library choice (unzipper, yauzl, or similar)
- Memory management for large JSON files within the ZIP
- Exact summary report JSON structure
- How to handle the `photos/` directory in the ZIP during Phase 2 (skip silently)
- PostGIS geography re-insertion approach (converting `{lat, lng}` back to `ST_MakePoint`)

</decisions>

## Existing Code Insights

### Reusable Assets
- `apps/admin/lib/auth.ts` — `requireAdmin()` for auth guard on import endpoint
- `apps/admin/lib/db.ts` — Prisma client with `$transaction()` support for atomic imports
- `apps/admin/app/api/export/route.ts` — TABLE_FILES array and ExportManifest type define the exact format the import must consume
- `apps/admin/app/dashboard/export/page.tsx` — Existing page to extend with import UI section

### Established Patterns
- Admin API routes: `apps/admin/app/api/{resource}/route.ts` with auth guard at top
- Response pattern: Direct `Response.json()` calls
- PostGIS: Export uses raw SQL `ST_Y/ST_X` — import will need raw SQL `ST_MakePoint` for the reverse
- Decimal fields: Export converts Prisma Decimal to `Number()` — import will write plain numbers back (Prisma handles conversion)

### Integration Points
- New API route: `apps/admin/app/api/import/route.ts` (POST handler, accepts ZIP upload)
- Extended UI: Import file picker and summary report added to existing export page
- Prisma `$transaction()` for atomic multi-table writes
- Raw SQL for PostGIS `ST_MakePoint` on Place inserts
